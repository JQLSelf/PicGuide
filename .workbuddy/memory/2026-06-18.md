## 文件夹模式"无图片"问题修复（深嵌套路径失效）

### 根因（双重叠加）
1. **SQLite LIKE 未指定 ESCAPE 子句** — `_escapeLike` 把 `_` 转成 `\_`，但 SQL 里 `\_` 在 SQLite 默认含义下：`\` 是字面字符 + `_` 是通配符 → 匹配"任意单字符"
   - 浅层路径 `D:\photos\cat.jpg` 不含 `_` → 无影响
   - 深嵌套路径 `D:\Users\john_doe\Pictures\...\img.jpg` → SQL 模式 `john\_doe` 匹配"johnX_doe"，**不匹配** `john_doe` → 整个文件夹被过滤为空
2. **`getDirectMediaInFolder` 截 substring 用错 prefix** — DB 里 filePath 可能是正斜杠，但代码用反斜杠 `prefix.length` 截，导致深嵌套路径下"相对路径"含分隔符 → 误判为子目录 → 全部过滤掉

### 修复内容
- `lib/src/db/database.dart`：
  - 5 处 `.like()` 改用 `_escapeLike` + `escapeChar: '\\'`（Drift 的 API 参数名是 `escapeChar` 不是 `escape`）
  - `getMediaInFolder` 加上路径归一化
  - `getDirectMediaInFolder` substring 改用 `actualPrefix`（按 filePath 实际分隔符选 prefix/prefixFwd）
  - `getItemsByCity` 加上 escape
  - `markMissingInFolder` 加上 escape
  - 给 `Tag` 加 `colorHex` 扩展（兼容旧代码）
- `lib/src/ui/browser/widget_search_bar.dart`：
  - 改 import：`provider_browser.dart`（旧/孤立/有错）→ `page_browser.dart` + `provider_app.dart`
- `lib/src/ui/browser/page_browser.dart`：
  - 增加 `distinctCamerasProvider` / `distinctCitiesProvider` 的 export
- **删除** `lib/src/ui/browser/provider_browser.dart`（孤立文件，重复定义 providers 且签名错乱）

### 验证
- `flutter analyze` 0 个 error，189 个 info 级别 lints（都是 `withOpacity`/`prefer_const_` 等风格提示）

### 教训
- Drift `Expression.like(String, {String? escapeChar})` — 参数名是 `escapeChar` 不是 `escape`
- 上一会话的"修复"实际未生效：grep 显示有 escape: '\\'，但文件实际没有（grep 把 `_escapeLike` 函数名也匹配上了）
- `provider_browser.dart` 与 `page_browser.dart` 存在重复 provider，但只 `widget_search_bar.dart` 引用了旧文件；删除旧文件前必须先迁移所有引用方

## 文件夹模式"深嵌套无图"bug 真正根因（2026-06-18）

### 症状
- 路径 `D:\XiaomiThemeEditor_v1.5.3_win_x64\resources\plugins\icons\iconCalendar\iconCalendar_src\icon_bg.png`
- 时间轴、标签模式：可见
- 文件夹模式：空白

### 调试过程
直接打开 SQLite 数据库（`build/windows/x64/runner/Debug/data/pixelvault.db`）：
1. 该记录**确实存在**：`is_deleted=0`, `is_missing=0`
2. id=179 实际存在且 thumbnail 已生成（3372 字节）
3. 测试 SQL：

| 查询 | 结果 |
|---|---|
| `LIKE 'D:\...\iconCalendar_src\%'` (无 ESCAPE) | **1 row** ✓ |
| `LIKE 'D:\...\iconCalendar_src\%' ESCAPE '\'` (我之前的"修复") | **0 rows** ✗ |
| `LIKE 'D:\..._v1.5.3\_win\_x64\%'` (用 _escapeLike 后的 pattern) | **0 rows** ✗ |

### 真正的根因
**我之前引入的"修复"反而把代码搞坏了。**

```dart
// 我之前"以为对的"修复：
t.filePath.like('${_escapeLike(prefix)}%', escapeChar: '\\')

// 实际生成的 SQL pattern:
// prefix = 'D:\XiaomiThemeEditor_v1.5.3_win_x64\resources\...\iconCalendar_src\'
// _escapeLike(prefix) = 'D:\XiaomiThemeEditor\_v1.5.3\_win\_x64\...\iconCalendar\_src\'
// 拼接 % → 'D:\XiaomiThemeEditor\_v1.5.3\_win\_x64\...\iconCalendar\_src\%'
//                                                            ↑ 外层 % 没 escape
//
// 加 ESCAPE '\' 后，\% = 字面 %（不是通配符！）
// 所以这个 pattern 实际要求 filePath 以字面 % 结尾 → 永远 0 行
```

### 关键教训
1. **SQLite `_` 通配符对字面 `_` 是匹配的**——通配符 = "任意单字符"，覆盖字面下划线
2. **加了 ESCAPE 子句后，**所有**作为通配符使用的 `%` 和 `_` 都必须 escape**——包括 pattern 末尾的 `prefix + '%'`
3. 不能局部 escape 然后让外层 % 保持通配符
4. **撤销"修复"反而是对的**：原始 `LIKE 'prefix%'`（无 escape、无 escapeChar）就能正确工作
5. 路径里有 `_` 不是问题：用户选 `D:\X_v1.5.3`，pattern `D:\X_v1.5.3\%` 里的 `_` 作为通配符，匹配字面 `_`（通配符包含字面字符）

### 修复
撤销所有 `_escapeLike` + `escapeChar` 改动，回到原始 `LIKE '$prefix%'`：

```dart
// getDirectMediaInFolder
t.filePath.like('$prefix%') |
t.filePath.like('$prefixFwd%')
```

5 处都改回：
- `getMediaInFolder`
- `getItemsByFolderPrefix`
- `getDirectMediaInFolder`
- `markMissingInFolder`
- `getItemsByCity`

### 真正的子目录过滤逻辑
`getDirectMediaInFolder` 里 `actualPrefix` 选 prefix 或 prefixFwd 的逻辑**仍然有用**——这跟 SQL 无关，是 Dart 字符串 substring 的对齐问题。这个保留。

### 验证
- `flutter analyze` 0 error
- 直接 SQL 测试：1 row（id=179）— 文件夹模式应该可以正确显示了

## PixelThumb 升级原图 bug 修复（2026-06-18）

### 症状
停留 5 秒后，"原图"角标出现了（说明 `_quality == full` 状态切了），
但图片本身**没变清晰**——还是 200x200 缩略图。

### 真因
`_imagePath()` 永远优先用 `thumbnailPath`（数据库里 200x200 的 JPG），
即使 `_quality == full` 也用它。
配合 `cacheWidth: null` 解码的是这张 200x200 的缩略图 → 看起来还是糊的。

### 修复
1. `_imagePath()` 改成：full 模式返回原文件，thumb 模式用缩略图
2. 加 `kFullCacheWidth = 1920` 常量（1080p 清晰度上限）
3. full 模式 `cacheWidth: 1920` 而非 `null`，避免 4000px 原图 OOM

### 内存管理：会清吗？

**会**，三层防御：

1. **Flutter ImageCache 自动 LRU**
   - `main.dart` 启动时配 `maximumSize: 300 项` + `maximumSizeBytes: 200 MB`
   - 超过任一上限自动驱逐最久未用的
   - 缩略图 + 升级原图 + 全屏图 共用同一个 LRU

2. **手动清理**（已实现）
   - `browserRefreshSignalProvider` 变化时（扫描/删除/编辑标签）→ `_AppShellState` 自动 `imageCache.clear()`

3. **widget 生命周期**
   - `ListView` / `GridView` 滚出视口 → 子 widget dispose → `Image.file` 不再持有引用
   - 但**像素仍可能在 ImageCache 里**（其他 widget 复用 key 之前）
   - `ImageCache` LRU 会在新图片进来时把旧的挤出去

**潜在风险**：
- 100 张图同时升级到 full（1920 宽）→ 峰值 200MB
- 加上全屏大图、缩略图，可能超 200MB 上限触发抖动
- **如果出现明显卡顿**，建议把 `kFullCacheWidth` 降到 1280，或把 `maximumSizeBytes` 调到 300 MB

### 验证
- `flutter analyze` 0 error
- 升级后图片会明显清晰（受限于原文件本身清晰度）

## 图片浏览器分页懒加载 + 时间轴预加载方案（评估 2026-06-18）

### 用户方案（接收后评估）

1. **先请求 20 页（建议改为"前 N 条"）**，拿总图片数 → 占位预渲染
2. **滚动 / 翻页 / 时间轴跳转** → 根据位置按需查询
3. **文件夹模式**：按当前目录查 → 同样 20 页
4. **时间轴**：扫描时存"yyyy-mm-dd 时间轴"表 → 懒加载时右侧时间轴也预加载
5. **维护时机**：扫描入库、软删除、对账时要同步时间轴表

### 评估：完全同意，方案合理。但有 5 个关键补充

#### 补充 1：明确"分页"是"分页渲染"还是"分页 DB 查询"
两种粒度：
- **只分页渲染**：DB 一次查 N 条（N=当前 sort 后的全量），UI 层分页渲染（ListView/GridView 本身已 lazy build）
- **分页 DB 查询**：DB 用 `LIMIT/OFFSET` 按需查

**建议**：先用方案 1（一次查全量）+ 已有的 lazy build 解决 80% 卡顿；方案 2 留作 P2。
- 理由：5 万条 MediaItem 记录，一次 `SELECT * FROM media_items` 在 SQLite 上只要 100-300 ms，不是瓶颈
- 真正的瓶颈是 **N 次 EXIF + Tag 关联查询**（`_enrichItem` 一次查 3 张表，Future.wait 5 万次 = 15 万次查询）

**真正要做的是**：把 `_enrichItem` 改成 **批量 JOIN**，一次查 3 张表：

```sql
SELECT m.*, e.*, GROUP_CONCAT(t.id) as tagIds
FROM media_items m
LEFT JOIN exif_datas e ON e.mediaItemId = m.id
LEFT JOIN media_tags mt ON mt.mediaItemId = m.id
LEFT JOIN tags t ON t.id = mt.tagId
WHERE m.isDeleted = 0
GROUP BY m.id
ORDER BY m.indexedAt DESC
```

#### 补充 2：时间轴预加载表的设计
建议用 **物化视图（materialized table）** 而非函数表：

```dart
@DataClassName('TimelineBucket')
class TimelineBuckets extends Table {
  TextColumn get bucketKey => text()();  // 'YYYY-MM'
  IntColumn get count => integer()();
  IntColumn get totalSize => integer().nullable()();  // 可选，统计用
  DateTimeColumn get updatedAt => dateTime().withDefault(currentDateAndTime)();
  
  @override
  Set<Column> get primaryKey => {bucketKey};
}
```

`bucketKey` = `yyyy-MM`（与现有 `_bucketKeyOf` 一致），主键保证 O(1) 查/写。

**不要**用"yyyy-MM-dd"——太细了，一个相册有几万条记录，右侧时间轴展开后会有几百行，反而难看。

#### 补充 3：维护策略
- **扫描入库时**：`onMediaInserted` → 反查 EXIF.dateTaken → 算 `bucketKey` → `INSERT OR IGNORE INTO timeline_buckets VALUES (key, 1) ON CONFLICT(bucketKey) DO UPDATE SET count = count + 1`
- **软删除时**：`onMediaSoftDeleted` → 同样逻辑，`count = count - 1`，归零时 `DELETE`
- **对账 missing**：`onMediaMissing` → 不影响时间轴（missing 文件的时间点用户可能还想看）
- **桶重新计算**：提供 `rebuildTimelineBuckets()`，全库扫一遍，onConflict 覆盖（用户改 EXIF 后用）

**关键**：不要每条插入/删除都跑 SQL——要批量（每 100 条提交一次），避免事务爆炸。

#### 补充 4：UI 层改造
```dart
// 旧的
final browserMediaProvider = FutureProvider<List<MediaItemWithMeta>>((ref) async {
  // 一次查全量 + N 次 _enrichItem
});

// 新的
final browserMediaPagedProvider = FutureProvider.family<
  ({List<MediaItemWithMeta> items, int totalCount, bool hasMore}),
  BrowserPageQuery

  // 一次 JOIN 查当前页 + 总数

```

`BrowserPageQuery` = `{ mode, folder?, tagId?, sort, offset, limit }`。

UI 层：
- **第一次进图库**：query = `{offset: 0, limit: 100}`（100 = 4 行 × 25 列）
- **滚动到底**：`ScrollEndNotification` 触发 → `query = {offset: 100, limit: 100}`，append 到列表
- **时间轴跳转**：从 `TimelineBuckets` 表读出 `bucketKey` 对应的首条 MediaItem 索引 → 直接用 `(bucketKey, offsetInBucket)` 做新 query

#### 补充 5：migration 与新表创建
新增 `TimelineBuckets` 表 → `schemaVersion: 3 → 4`：

```dart
@override
MigrationStrategy get migration => MigrationStrategy(
  onCreate: (m) => m.createAll(),
  onUpgrade: (m, from, to) async {
    if (from < 2) { ... }
    if (from < 3) { ... }
    if (from < 4) {
      await m.createTable(timelineBuckets);
      // 全库回填一次
      await customStatement('''
        INSERT INTO timeline_buckets (bucket_key, count, updated_at)
        SELECT strftime('%Y-%m', COALESCE(e.date_taken, m.file_modified_at, m.indexed_at)) AS bk,
               COUNT(*), CURRENT_TIMESTAMP
        FROM media_items m LEFT JOIN exif_datas e ON e.media_item_id = m.id
        WHERE m.is_deleted = 0
        GROUP BY bk
        ORDER BY bk DESC
      ''');
    }
  },
);
```

### 实施步骤（如果要做）

### 不建议做的

- **不要**用"20 页"作为分页粒度（page 在不同模式大小不同）；用 "limit = 100 条" 更直观
- **不要**给每个文件夹都建子时间轴（文件夹模式只看 direct 媒体，没有"时间轴"概念）
- **不要**给 TimelineBuckets 加索引（bucketKey 是主键已经够快）
- **不要**用 virtual scroll（flutter_staggered_grid_view 的 `MasonryGridView` 不支持；硬上 ListView 会失去瀑布流效果）

### 风险点

- 软删除/对账 missing 时维护 TimelineBuckets → 必须在 service 层强制走 `upsertMediaItem` / `softDeleteMedia` 才能保证同步
- 大量历史数据回填：5 万条数据 `GROUP BY strftime('%Y-%m', ...)` 在 SQLite 上约 1-2 秒，迁移时 UI 会卡 1-2 秒；可延后到 `addPostFrameCallback` 后台跑
- 时间轴点击跳转的精度：`bucketKey` 只能定位到月份首项，不能定位到具体日期首项——用户点"2024-01"会跳到该月第一张照片，但月份内有 100 张和 1000 张体验不同
